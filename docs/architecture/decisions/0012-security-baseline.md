# ADR-0012: Security baseline for Phase 1

- **Status:** Accepted
- **Date:** 2026-05-09
- **Decider:** Product owner (CEO)

## Context

Modern software delivery has converged on a small, well-understood security baseline that most professional teams meet from the start of a project. The baseline includes static analysis of source code (SAST), software composition analysis to catch known vulnerabilities in dependencies (SCA), secrets scanning to catch accidentally committed credentials, and supply-chain hygiene practices including pinned action versions in CI and software bills of materials. None of these practices is exotic in 2026, and each has well-documented tooling that integrates with GitHub Actions in minutes rather than days.

For a project being built primarily by an AI coding agent, the security baseline is more important than usual rather than less. Two patterns documented in the practitioner literature on AI-assisted development bear directly on FleetCo. The first is that AI-generated code is more prone than human-written code to including dependencies that look plausible but do not exist (or that exist and are malicious squatters on plausible names). The second is that AI agents are more prone than humans to embedding plausible-looking secrets, sample API keys, or hardcoded credentials in code, particularly when generating example configurations or test fixtures. Both patterns are caught by the standard security baseline, which makes the baseline a direct mitigation for known agent failure modes rather than a generic best practice.

For FleetCo specifically, the data sensitivity argues for adopting the baseline early rather than late. Even in Phase 1 the system holds driver names and license numbers, vehicle registration data, customer commercial terms, and the founder's own administrative credentials. Phase 2 adds GPS traces, which are personally identifying data of a particularly sensitive kind (a GPS trace of a heavy-construction vehicle reveals the work patterns of the company and the location patterns of the driver). The cost of a credential leak or a known-vulnerability exploit is substantially higher in this context than in a typical pre-revenue bootstrap.

## Decision

We commit to a minimum viable security baseline that runs on every PR from the moment Phase 0 ends. The baseline has five components, all of them GitHub Actions jobs that run as part of the standard CI pipeline.

The first component is dependency scanning via Dependabot, configured to alert on known vulnerabilities in npm dependencies and to automatically open PRs for security-relevant version bumps. Dependabot is configured at the repository settings level and requires no additional CI work; it simply needs to be enabled.

The second component is secrets scanning via GitHub's native secrets scanning, which runs on every push and on the full git history. GitHub's secrets scanning catches a known set of credential patterns (AWS keys, Stripe keys, GitHub tokens, and many others) and is enabled at the repository level. The bootstrap also commits to never including a real secret in any committed file, and the PR template enforces this with a checkbox that asks the contributor to confirm that no secrets are present.

The third component is static analysis via Semgrep with the OWASP and security-audit rulesets. Semgrep runs as a CI job on every PR and fails the build if it finds high-severity issues. Semgrep is configured via a workflow file in `.github/workflows/security.yml` that the Phase 0 task list includes.

The fourth component is action-version pinning. Every GitHub Actions workflow uses pinned commit SHAs for third-party actions rather than version tags or branch names. This protects against the supply-chain attack pattern in which a malicious actor takes over a popular GitHub Action and pushes a compromised version under an existing tag. Pinned SHAs mean that even if the upstream tag is moved, the workflow continues to use the version we audited. Renovate or Dependabot is configured to propose updates to these SHAs so that pinning does not become stagnation.

The fifth component is software bill of materials generation, which produces a CycloneDX SBOM as a CI artifact for every production deployment. The SBOM is generated by `cyclonedx-bom` for the npm workspace and is uploaded to the GitHub Actions release artifacts. This is the lightest-weight version of supply-chain provenance and corresponds approximately to SLSA Level 1.

We deliberately do not adopt heavier security tooling at this stage. We do not adopt container or image scanning (because we are not yet using containers in production), we do not adopt infrastructure-as-code scanning (because we have not yet codified the infrastructure-as-code), and we do not adopt signed-artifact provenance via Sigstore or Cosign (because the operational benefits at our scale do not yet justify the integration cost). These are deferred to a future ADR that will be written when the relevant substrate exists.

## Alternatives considered

Skipping the security baseline until after Phase 1 ships was considered and rejected. The cost of adding the baseline before any code is written is a few hours of CI configuration. The cost of retrofitting it after the codebase has grown is substantially higher, both because the existing code may have accumulated issues that the baseline would catch and because the retrofit interacts with whatever development patterns have settled into place. The early adoption is the cheap path.

Adopting a more elaborate baseline (commercial SAST tools, container scanning, signed artifacts, full SLSA Level 2 or higher) was considered and rejected as premature for the bootstrap. Each of these tools has real value at the scale of organizations with dedicated security teams; for a one-person bootstrap they would consume more attention than they return. The right move is to adopt the minimum baseline now and to revisit each tool individually when its specific value becomes load-bearing.

Adopting a different SAST tool (CodeQL, SonarCloud, Snyk Code) instead of Semgrep was considered. Semgrep was chosen because its rulesets are open and human-readable, its integration with GitHub Actions is mature, and its false-positive rate on the languages we use (TypeScript, with light Python for tooling) is well-characterized in the practitioner literature. CodeQL would also be defensible; the choice between them is mostly aesthetic at this scale.

## Consequences

The baseline costs a few minutes per PR in CI time and adds approximately one new failure mode (a security finding that needs to be addressed before the PR can merge). The benefit is that the standard agent failure modes (hallucinated dependencies, accidental secrets, outdated and vulnerable packages) are caught at the merge boundary rather than after they ship. The framework also gives us a defensible answer if a customer or investor ever asks about our security posture: we have a documented baseline, we have CI evidence that the baseline runs, and we have an SBOM for every production deployment.

The PR template is updated to include a security checklist: no secrets present, no known-vulnerable dependencies introduced, action SHAs pinned, SBOM generated. These checks are partly mechanized by the CI but listing them explicitly in the template ensures that the human reviewer notices when one of them fires.

The runbook acquires a new section on security incident response, which is a procedure for handling a credential leak, a vulnerability disclosure, or a CI security finding that cannot be fixed immediately. The procedure references this ADR.

## Revisit when

The signal to expand the security baseline is the introduction of containerized deployment (which adds image scanning), the codification of infrastructure-as-code (which adds IaC scanning), or the addition of customer-facing artifacts that benefit from cryptographic provenance (which adds Sigstore and SLSA Level 2). The signal to revisit the choice of tools is when one of them produces a sustained pattern of unhelpful output (false positives, low-signal alerts) that the team is ignoring; an ignored security tool is worse than no security tool because it teaches the habit of dismissing security alerts.
