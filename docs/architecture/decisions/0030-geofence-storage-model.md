# ADR-0030: Geofence storage model — a Tier-3 polygon aggregate, hybrid PostGIS storage, and the read/write RBAC split

- **Status:** Proposed
- **Date:** 2026-06-04
- **Decider:** Product owner (CEO) — pending acceptance

## Context

Phase 2's GPS telematics ingestion program (ADR-0029, tickets T1–T5) shipped end-to-end. Its last slice (T5) built the geofencing **query** — `ST_Contains` / `ST_DWithin` over the `GpsPing.geometry` generated Point column — but it runs against a **caller-supplied** boundary passed as query parameters: a circle (`centerLatitude`/`centerLongitude`/`radiusMeters`) or a polygon (`polygon=lon,lat;lon,lat;…`, parsed by `apps/api/src/modules/telematics/telematics.schemas.ts` into a closed `POLYGON((…))` WKT ring, 3–1000 vertices, auto-closed). **There is no stored geofence.** ADR-0029 (T5 sketch in commitment 13) and the glossary's *Geofence* entry both explicitly deferred geofence-polygon **storage** to "its own sibling slice," noting the T5 query is written so that swapping a query-param WKT for a stored row's geometry is "~a one-line change."

This ADR is that sibling slice's decision record. Geofencing for the depot, customer sites, and route corridors is a named Phase-2 deliverable (`docs/product/roadmap.md` §"Phase 2"). To make it real, the system needs a **first-class, manageable `Geofence` aggregate**: a place to define and edit the boundaries the telematics read layer classifies vehicles against. That aggregate introduces FleetCo's **first PostGIS `geometry(Polygon, 4326)`** (T2 stored the first Point), a new operational-configuration data model, and new access-control surface — exactly the schema + cross-cutting profile CLAUDE.md step 4 wants confirmed via an ADR before code, mirroring ADR-0028 (RBAC model before RBAC code) and ADR-0029 (ingestion architecture before GPS code). This ADR **writes no code**; it fixes shape and policy, and the geofence code program builds the commitments.

The substrate this builds on already exists and constrains the choices:

- **The hybrid PostGIS pattern (ADR-0029 c8, migration `apps/api/prisma/migrations/20260604080923_add_gps_ping/migration.sql`).** `GpsPing` stores native `Float` `latitude`/`longitude` (Prisma-native, canonical) plus a `geometry(Point, 4326)` column declared `Unsupported("geometry(Point, 4326)")?` in `schema.prisma` and defined in hand-authored migration SQL as `GENERATED ALWAYS AS (ST_SetSRID(ST_MakePoint("longitude","latitude"), 4326)) STORED` with a GIST index. Prisma writes only the Floats; the database derives the geometry losslessly so it cannot drift; raw SQL is confined to spatial predicates. The migration is hand-authored (`--create-only` then edited; never edited after merge), and a round-trip test guards the `ST_MakePoint(lon, lat)` X,Y foot-gun. **A polygon, however, is not derived from scalar floats** — which is the one structural difference this ADR must resolve (commitment 1).
- **The aggregate CRUD pattern (`apps/api/src/modules/customers/`).** Every Phase-1 aggregate follows: `model` + migration → service (`list`/`findById`/`create`/`update`/`delete`) → controller (`@UseGuards(AuthGuard)`, Zod `.strict()` schemas, the `{ items, total, skip, take, sortBy, sortDir }` list shape, the `Cuid` helper, `createdById` from `request.session.user.id`) → tests in `apps/api/test/` (real Postgres+PostGIS) → web pages under `apps/web/src/app/<aggregate>/` + a nav button. Prisma error mapping is standardized (`docs/runbook/api-error-mapping.md`): P2002→409, P2003→409 on delete / 400 on insert, P2025→404. **`CustomersService.delete` already carries a live `P2003 → 409` catch arm** ("Cannot delete customer: it is referenced by other records."), so a new FK into `Customer` is just another referencer it already covers.
- **The RBAC capability map (ADR-0028, `apps/api/src/modules/auth/permissions.ts`).** A closed `Capability` union + a hardcoded `ROLE_CAPABILITY_MAP` keyed by `UserRole`. A shared `OPERATIONAL_CAPABILITIES` floor (held by both ADMIN and OFFICE_STAFF) already includes `gps:read-derived` (the live-location map / geofence status / route summary); ADMIN layers sensitive tokens (`gps:read-raw`, `users:manage`, …) on top. Adding a capability is a **map edit**, and the closed union makes a typo a compile error. The geofence-**status** read is already gated `gps:read-derived`; what's new here is the capability for managing the geofence **configuration** aggregate itself.
- **The Tier-3 classification (ADR-0027 c6).** Geofence polygons (depot, customer site, route corridor) are **Tier 3 operational business data** — company configuration assets, not the Tier-5 surveillance trail they are checked against. ADR-0027 c6 fixed this; this ADR consumes it.

This ADR **decides shape and policy only** — it writes no model, no migration, no controller, no capability-map edit, no UI. Those land with the geofence code program that consumes the decisions fixed here; the code waits for acceptance.

## Decision

**Store geofences as a single Tier-3 `Geofence` aggregate with a `GeofenceType` enum (DEPOT / CUSTOMER_SITE / ROUTE_CORRIDOR), using a hybrid representation that extends ADR-0029's pattern to polygons — a Prisma-native canonical `boundaryWkt` text column plus a `GENERATED ALWAYS AS (ST_GeomFromText("boundaryWkt", 4326)) STORED geometry(Polygon, 4326)` column with a GIST index — with the canonical boundary expressed in the exact `lon,lat;…` vertex representation T5 already parses (so a stored fence and an ad-hoc query-param fence are byte-identical and interchangeable); enforce validity in three layers (Zod vertex checks, the generated column's parse-time rejection, and a service-level `ST_IsValid` gate against self-intersecting rings); own customer-site fences via a nullable `customerId` FK (`onDelete: Restrict`, already covered by Customer's live 409 arm) with a type/ownership refine; gate the new aggregate with a `geofences:read` capability (ADMIN + OFFICE_STAFF) and a `geofences:write` capability (ADMIN-only); defer stored geofence-crossing events to a later worker slice; and ship a v1 with a structured coordinate-entry UI (no map-drawing dependency).** Eight commitments define the shape. As with ADR-0028 and ADR-0029, the genuinely owner-level picks — the storage representation, the read/write RBAC split, and the no-map-v1 dependency posture — are left to the PO at acceptance; the rest are recorded recommendations.

### A. Storage and validity

1. **Polygon storage: extend the hybrid — a canonical `boundaryWkt` text column + a generated `geometry(Polygon, 4326)` (recommended).** *(Owner-level pick.)* `ST_GeomFromText(text, srid)` (the two-argument form with an explicit SRID) is **`IMMUTABLE`** in PostGIS, and Postgres `GENERATED ALWAYS AS (…) STORED` columns require an immutable expression — so the GpsPing hybrid extends cleanly to polygons. The `Geofence` row stores a Prisma-native `boundaryWkt` **text** column (canonical, written and read by Prisma) plus a `geometry Unsupported("geometry(Polygon, 4326)")?` column defined in hand-authored migration SQL as `GENERATED ALWAYS AS (ST_GeomFromText("boundaryWkt", 4326)) STORED`, with a GIST index. This is the Point pattern with one structural change stated plainly: **the polygon geometry derives from a text column, not from scalar floats** (a polygon has no scalar canonical form). It keeps inserts/reads on the type-safe Prisma client, confines raw SQL to spatial predicates, and makes the geometry undriftable because the database derives it. The **canonical boundary uses the exact `lon,lat;lon,lat;…` vertex representation `telematics.schemas.ts` already parses** into a closed `POLYGON((…))` WKT ring — reusing that builder means a **stored fence and a T5 query-param fence are byte-identical WKT** and classify identically, which is the whole point of the feature. **The alternative the PO may choose:** a **geometry-only** `Unsupported` column with no canonical text, inserted via `$executeRaw ST_GeomFromText(...)` — the honest fallback if the CI-pinned PostGIS build ever does not accept `ST_GeomFromText` in a generated context (the code slice verifies immutability against the pinned version before merging, exactly as ADR-0029's turn-on verifies its exporter path). Geometry-only costs the Prisma-native insert path (every geofence write becomes raw SQL), acceptable on a low-volume config table but dispreferred for consistency with ADR-0029.

2. **Validity: three layers, Zod-first, with an `ST_IsValid` gate for self-intersection.** A stored fence is durable configuration, so it earns stricter validation than T5's ephemeral query fence. (a) **Zod** reuses T5's vertex parser — 3–1000 vertices, finite lon/lat in WGS84 range, auto-closing the ring — rejecting malformed input as HTTP 400 with the same messages. (b) **The generated column** rejects structurally-broken WKT at insert: `ST_GeomFromText` *throws* (it does not yield NULL), which the service maps to a clean 400/422 (a raw geometry-parse error, not a Prisma error code). (c) **A service-level `ST_IsValid` gate** (`SELECT ST_IsValid(ST_GeomFromText($1, 4326))` before the write) rejects **self-intersecting / bowtie** rings — structurally-valid WKT that `ST_GeomFromText` accepts but `ST_Contains` misclassifies. T5 never needed this because an ad-hoc query fence was throwaway; a stored fence that silently misclassifies forever is a real defect, so the gate is load-bearing.

### B. Data model

3. **Type taxonomy: one `Geofence` table + a `GeofenceType` enum {DEPOT, CUSTOMER_SITE, ROUTE_CORRIDOR}.** All three geofence kinds share an identical structure (name, boundary, ownership, audit) and differ only in semantics and ownership, so a single table with a type discriminator is the right shape — consistent with the `CustomerStatus` / `ExpenseCategory` single-table-plus-enum convention, and avoiding the triplicated CRUD/controller/web surface of per-type tables. Cardinality is left to operational reality: the schema does **not** enforce a singleton depot (a fleet may run more than one yard), and adding a fourth type later is a one-line enum migration.

4. **Ownership: a nullable `customerId` FK (`onDelete: Restrict`) + a type/ownership refine.** A CUSTOMER_SITE fence belongs to a `Customer` (a customer-site boundary is already Tier-3 customer data per ADR-0027 c6); DEPOT and ROUTE_CORRIDOR fences have `customerId = null`. `onDelete: Restrict` matches the house posture (GpsPing, FuelLog, the audit FKs all Restrict), and **`CustomersService.delete` already has the live `P2003 → 409` arm**, so deleting a customer that owns a fence returns a clean 409 with no change to `CustomersService` — the new referencer is covered for free. A Zod `.superRefine` enforces the type/ownership invariant (CUSTOMER_SITE requires non-null `customerId`; DEPOT/ROUTE_CORRIDOR require null), mirroring T5's circle-XOR-polygon refine. The dispreferred alternative is `SetNull` — silently orphaning a customer-site fence to an ownerless one is worse than a 409 the operator resolves deliberately, and is inconsistent with the house Restrict posture.

### C. Access control

5. **RBAC: a `geofences:read` capability (ADMIN + OFFICE_STAFF) and a `geofences:write` capability (ADMIN-only).** *(Owner-level pick.)* Managing geofences is the management surface for the boundaries the derived views read against, so the read/write split mirrors ADR-0028's privilege reasoning. **`geofences:read`** joins `OPERATIONAL_CAPABILITIES` in `permissions.ts` (both live roles) because OFFICE_STAFF need to see fences for the live-location map and geofence-status views — exactly parallel to the `gps:read-derived` token already on that floor. **`geofences:write`** (create / update / delete the configuration) joins the **ADMIN set** only, because geofence boundaries are operational *configuration* at the same privilege tier as `users:manage` — an OFFICE_STAFF session should read fences, not redraw the depot boundary. The verb-token style (`:read` / `:write`) matches the existing `gps:read-derived` / `gps:read-raw` / `reports:read` tokens rather than the coarse operational `<aggregate>:*` tokens, precisely so the two operations carry different privilege. Both additions are map edits in `permissions.ts`; the closed `Capability` union makes a typo a compile error. The dispreferred alternative — a single coarse `geofences:*` operational token on the shared floor — is rejected because it would let OFFICE_STAFF rewrite fence configuration, collapsing the split this commitment exists to create.

### D. Scope boundaries

6. **Derived geofence-crossing events are out of scope (deferred).** This slice is **storage + wiring the existing on-demand status query** to read a stored boundary. A stored stream of crossing events ("vehicle X entered the depot geofence at 14:32") is a separate future worker slice — it needs a queue job evaluating each incoming ping against the stored fences, an evaluation cadence, and falls squarely under ADR-0027 c6's anti-circumvention clause (a crossing event per ping is dense enough to reconstruct the raw Tier-5 trail and would *stay Tier 5*). Keeping it out keeps this slice a tight, independently-shippable vertical, exactly as ADR-0029 kept geofence storage out of its read slice. It is named in "Revisit when" as the next sibling.

7. **Classification: Tier 3, no new egress controls.** `name`, `boundaryWkt`, `type`, and `customerId` carry `/// Tier 3 per ADR-0013` marks — operational configuration, like a Job record. **No** GPS-style pino-`redact` or span-scrub entries are added: a configured boundary is a company asset, not the Tier-5 trail of a named driver, and over-redacting configuration hides operational signal. One clarification this ADR makes explicit, because ADR-0027 c5's provisional scrub denylist listed "any geofence polygon geometry": that hedge guarded against a polygon **co-traveling inside a Tier-5 telematics span**, not against the configuration table — a static, company-defined `Geofence` row is Tier-3 and is neither logged-as-secret nor scrubbed-from-spans the way raw coordinates are. (If a future span ever carries a polygon alongside a vehicle's live position, that span's scrubbing is governed where the span is built, not here.)

### E. Surface

8. **v1 ships a structured coordinate-entry UI — no map-drawing dependency (recommended).** *(Owner-level pick.)* A map-based polygon-drawing editor (Leaflet / MapLibre + a draw plugin) is a new top-level frontend dependency, which CLAUDE.md requires proposing for confirmation. v1 instead ships a **coordinate-entry form** accepting the same `lon,lat;lon,lat;…` representation the API already parses (commitment 1) — adequate to define a depot or site boundary, zero new dependency, and it keeps the storage slice tight. A map-based draw/edit editor is a **named follow-up slice** that proposes its mapping library on its own merits. The alternative — shipping the map library now — is dispreferred for v1 to avoid coupling a new dependency to the storage slice; the PO may choose to bring it forward.

### Follow-up code-slice sequence (proposed, not executed)

This ADR sketches the path so the PO sees it; it executes none of it. Each ticket is a vertical, independently-shippable slice:

- **G1 — `Geofence` schema + migration.** `model Geofence` (commitment 3 fields, the hybrid storage of commitment 1, the nullable `customerId` FK of commitment 4, Tier-3 marks of commitment 7, indexes on `type`/`customerId`/`createdById`) + the `Customer.geofences` back-relation; the hand-authored generated-column + GIST migration SQL; the `ST_GeomFromText`-immutability check resolved against the CI-pinned PostGIS (it gates G2's insert path); a round-trip schema test mirroring `gps-ping.schema.test.ts` (SRID 4326, stored ring == input WKT, self-intersecting ring rejected).
- **G2 — service + controller + Zod schemas + RBAC + tests.** Full CRUD mirroring `customers/`; reuse T5's WKT ring builder (commitment 1); the `ST_IsValid` gate and type/ownership refine (commitments 2, 4); `geofences:read` / `geofences:write` added to `permissions.ts` (commitment 5) with the controller gated by `RolesGuard` + `@RequirePermission`; tests mirroring `customers.{service,controller}.test.ts` plus the RBAC split (OFFICE_STAFF 403 on write) and a customer-delete-blocker 409.
- **G3 — web admin UI.** `apps/web/src/app/geofences/` (list / detail / new / edit / delete + `actions.ts` + nav button), the coordinate-entry form of commitment 8, a customer picker shown only for CUSTOMER_SITE — mirroring `apps/web/src/app/customers/`.
- **G4 — wire the stored fence into the T5 status query.** Add a `?geofenceId=…` branch to `GeofenceStatusQuerySchema` (a three-way XOR: circle / polygon / geofenceId), load the stored fence (404 if missing), and run the **existing** `ST_Contains(ST_GeomFromText("boundaryWkt", 4326), "geometry")` — the "one-line change" ADR-0029 T5 anticipated; the gate stays `gps:read-derived`. A test asserts a stored fence and an equivalent query-param polygon classify identically.

### Relationship to prior ADRs

- **Consumes ADR-0029** — extends its hybrid PostGIS storage (c8) from Point to Polygon, reuses its `Unsupported`/generated-column/GIST/no-drift migration discipline, and wires the stored fence into the geofence-status query its T5 built against query params. ADR-0029 is not edited.
- **Consumes ADR-0027** — its c6 Tier-3 classification of geofence polygons is the input; this ADR clarifies (not amends) the c5 "geofence polygon geometry" scrub mention as span-scoped, not config-scoped.
- **Consumes ADR-0028** — reuses the `ROLE_CAPABILITY_MAP` + `RolesGuard` + `@RequirePermission` mechanism and the `OPERATIONAL_CAPABILITIES` floor; adds two capabilities as map edits.
- **Is the PO-confirmation vehicle** for the first PostGIS Polygon and the new configuration aggregate, per CLAUDE.md step 4 — no `Geofence` code is written until this ADR is accepted.
- **On acceptance**, the glossary's *Geofence* / *PostGIS* entries gain a pointer to this ADR and `docs/CURRENT_PHASE.md` notes the geofence model is set — those memory updates land with the implementing slices, not in this proposal.

## Alternatives considered

**Geometry-only column, inserted via raw SQL (no canonical text).** The cleanest single-source-of-truth representation, and the documented fallback if `ST_GeomFromText` is ever not usable in a generated column on the pinned PostGIS. Dispreferred because it forfeits the Prisma-native insert/read path the hybrid keeps, making every geofence write raw SQL — an inconsistency with ADR-0029's hybrid for no benefit at this table's low volume. Kept available as the PO's call (commitment 1).

**Store GeoJSON instead of the `lon,lat;…` vertex list.** GeoJSON (RFC 7946) is a richer, standard interchange format. Rejected as the canonical input because the system already parses, validates, and tests the `lon,lat;…` vertex representation in `telematics.schemas.ts`, and reusing it guarantees a stored fence and a query-param fence are identical. GeoJSON can be accepted as an additional input format later by converting to the same WKT; it is not the canonical store.

**Per-type tables (Depot / CustomerSite / RouteCorridor) instead of one enum-typed table.** Rejected (commitment 3): the three share identical structure and differ only in ownership/semantics, so separate tables triplicate the CRUD/controller/web surface and break the established enum convention for no schema divergence.

**A single coarse `geofences:*` operational capability.** Mirrors the `customers:*` operational tokens most literally and is the simplest map edit. Rejected (commitment 5) because it grants both roles full CRUD, collapsing the read/write privilege split — an OFFICE_STAFF session should read fences for the operational views but not redraw the depot boundary.

**Ship the map-drawing editor in v1.** The better end-state UX. Rejected for v1 (commitment 8) only to avoid coupling a new frontend mapping dependency to the storage slice; it is a named follow-up that proposes its library on its own. This is a sequencing choice, not a rejection of a map UI.

**Build geofence storage without a separate ADR.** Rejected on CLAUDE.md step 4: the first PostGIS Polygon, a new configuration aggregate, and new access-control tokens are exactly the schema + cross-cutting changes that warrant PO confirmation, mirroring ADR-0028 and ADR-0029 — so the PO ratifies one coherent geofence model rather than discovering it in feature PRs.

## Consequences

### What this makes easier

The geofence code program arrives with its model settled: a known storage representation that extends the pattern already in the repo, a known type taxonomy and ownership rule, a known validity-enforcement stack, a known RBAC split, and a four-ticket roadmap. The T5 geofencing query — already written against a WKT polygon — gains a stored source with a one-line change, completing the depot/customer-site/route-corridor geofencing the roadmap named. Because the storage reuses ADR-0029's hybrid and the CRUD reuses the Customers pattern, the slice is mostly composition of proven parts. And the geofence model becomes legible memory: a future agent reads one ADR and knows how a boundary is stored, validated, owned, and who may edit it.

### What this makes harder

A second PostGIS storage idiom enters the schema (a polygon derived from text, alongside the point derived from floats) — a contributor must know a geofence geometry comes from `boundaryWkt`, not from coordinates. The hand-authored generated-column migration and its no-drift discipline recur. A self-intersecting ring is a new validity hazard the `ST_IsValid` gate must hold. And the representation-coherence guarantee (stored and query-param fences identical) is a standing obligation: the geofence store and the T5 query must keep using the same WKT builder, or the two paths drift.

### Costs we accept

- **No map UI in v1.** Defining a polygon by typing `lon,lat` pairs is serviceable but not pleasant; the map editor is deferred. We accept manual coordinate entry until the follow-up slice proposes a mapping library.
- **The hybrid leans on `ST_GeomFromText` immutability.** If a future PostGIS changes that, the generated column breaks and the store flips to the raw-insert fallback. The code slice pins and verifies immutability at G1, keeping the blast radius to one ticket.
- **Crossing events are deferred.** Until the worker slice lands, geofence membership is computed on demand at read time, not recorded as a queryable event history; some "when did vehicle X enter the depot?" questions wait for that slice.

## Revisit when

- **PO acceptance fixes the open picks.** This ADR is Proposed; acceptance ratifies the storage representation (commitment 1, hybrid vs geometry-only), the read/write RBAC split (commitment 5), and the no-map-v1 dependency posture (commitment 8). If the PO chooses differently from a recommendation, the code slice follows the PO's pick and this ADR is annotated accordingly.
- **Stored geofence-crossing events are wanted (commitment 6).** When the PO wants a queryable enter/exit history, that worker slice lands — governed by ADR-0027 c6's Tier-3 classification and anti-circumvention clause.
- **A map-drawing editor is wanted (commitment 8).** The follow-up UI slice proposes its mapping library (Leaflet / MapLibre + draw plugin) as a new dependency for confirmation.
- **GeoJSON or another interchange format is needed** (e.g. importing boundaries from an external GIS) — add it as an additional input that converts to the canonical WKT store, not as a second source of truth.
- **The fleet outgrows a single depot / simple polygons** — revisit whether route corridors want `geometry(LineString)` + buffering rather than hand-drawn polygons, and whether multi-depot topology needs scoping.
- **Prisma gains native PostGIS support** (select/insert of `geometry`) — revisit the hybrid, as ADR-0029 c-"Revisit when" already notes; the move would be mechanical and shared with GpsPing.
